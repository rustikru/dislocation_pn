# Модуль ГУ-23 — Акты общей формы

Веб-модуль для составления, регистрации и электронного согласования актов общей формы ГУ-23.

## Схема базы данных

### Основные таблицы

| Таблица                 | Назначение                                       |
| ----------------------- | ------------------------------------------------ |
| `xx_disl_gu23_act`      | Заголовок акта (тип, статус, даты, цех, станции) |
| `xx_disl_gu23_act_row`  | Строки акта (вагоны)                             |
| `xx_disl_gu23_signer`   | Подписанты конкретного акта                      |
| `xx_disl_gu23_approval` | Запросы согласования (pending/approved/rejected) |
| `xx_disl_gu23_hist`     | История изменений акта                           |
| `xx_disl_gu23_file`     | Прикреплённые файлы/фото                         |
| `xx_disl_gu23_counter`  | Счётчик нумерации (по цеху и году)               |

### Справочники

| Таблица                   | Назначение                            |
| ------------------------- | ------------------------------------- |
| `xx_disl_gu23_ref_reason` | Причины составления акта              |
| `xx_disl_gu23_ref_signer` | Справочник подписантов от стороны РЖД |

> **Подписанты от предприятия** хранятся в `xx_disl_users`.  
> **Подписанты от РЖД** хранятся в `xx_disl_gu23_ref_signer` (не имеют учётных записей и не подписывают электронно).

### Поле `stype` в `xx_disl_gu23_signer`

Добавлено миграцией `migrate_signer_stype.sql`.

| Значение | Смысл                                                                 |
| -------- | --------------------------------------------------------------------- |
| `'own'`  | Работник предприятия (`signer_ref_id` → `xx_disl_users.id`)           |
| `'rzd'`  | Работник станции РЖД (`signer_ref_id` → `xx_disl_gu23_ref_signer.id`) |

Поле сохраняется фронтендом при выборе подписанта из соответствующего выпадающего списка — это единственный надёжный способ различить типы, т.к. ID в обеих таблицах могут совпадать.

### Представление `xx_disl_gu23_act_v`

Объединяет шапку акта с вычисляемыми полями: номер связанного акта начала, количество вагонов, количество файлов. Пакет читает акты только через это представление.

## Жизненный цикл акта

```
draft → active → closed      (для актов типа 'end', только администратором)
              ↘ annulled     (для 'active' и 'closed', только администратором, с каскадом)
```

| Статус     | Описание                             |
| ---------- | ------------------------------------ |
| `draft`    | Проект, не зарегистрирован           |
| `active`   | Зарегистрирован, имеет номер         |
| `closed`   | Закрыт (только для актов типа `end`) |
| `annulled` | Аннулирован                          |

### Типы актов

| Тип     | Описание                                     |
| ------- | -------------------------------------------- |
| `start` | Начало простоя                               |
| `end`   | Окончание простоя (ссылается на `start`-акт) |
| `other` | Прочий акт                                   |

### Каскадное аннулирование

При аннулировании акта типа `end` автоматически аннулируется связанный акт начала (`linked_start_id`), и наоборот.

## Oracle-пакет `xx_disl_gu23_pkg`

Весь доступ к данным идёт только через пакет. Прямые DML из PHP запрещены.

### Основные функции/процедуры

| Имя                         | Описание                                                    |
| --------------------------- | ----------------------------------------------------------- |
| `gu23_get_refs`             | Справочники (цеха, станции, причины, подписанты)            |
| `gu23_get_acts`             | Реестр актов с фильтрами                                    |
| `gu23_get_act`              | Карточка одного акта                                        |
| `gu23_save_act`             | Сохранение / регистрация акта                               |
| `gu23_annul_act`            | Аннулирование (с каскадом)                                  |
| `gu23_close_act`            | Закрытие акта типа `end` (только `active` → `closed`)       |
| `gu23_get_signers`          | Подписанты акта (с полем `stype`)                           |
| `gu23_approval_init`        | Создание pending-записей согласования                       |
| `gu23_approval_get_signers` | Подписанты предприятия (`stype='own'`) с email для рассылки |
| `gu23_direct_decision`      | Подписание/отклонение в приложении (пишет историю)          |

## PHP-слой (`GuActRepository.php`)

### `handle(string $action)`

Диспетчер: принимает `ajax_action` из POST и вызывает нужный приватный метод.

### Вспомогательные методы

**`pipe(string $sql, array $binds): array`**  
Выполняет SELECT и возвращает массив строк. Используется для простых запросов и чтения данных из пакетных функций, возвращающих таблицу.

**`callFunc(string $expr, array $binds, int $retLen): ?string`**  
Вызывает скалярную PL/SQL-функцию через `BEGIN :ret := expr; END;`. Бросает `RuntimeException` при ошибке OCI (не выводит HTML-варнинги, не ломает JSON).

**`packRows(array $rows, array $fields): string`**  
Упаковывает массив строк в строку с разделителями `CHR(30)` (между записями) и `CHR(31)` (между полями) для передачи в пакет через CLOB.

### Флаги в ответе `gu23_get_act`

| Поле           | Тип    | Описание                                                                                |
| -------------- | ------ | --------------------------------------------------------------------------------------- |
| `isAdmin`      | bool   | Текущий пользователь — администратор                                                    |
| `isUserSigner` | bool   | Текущий пользователь есть среди подписантов акта (`stype='own'`)                        |
| `myApproval`   | string | Статус согласования текущего пользователя: `pending` / `approved` / `rejected` / `none` |

## Согласование актов

### Инициализация (`gu23_send_approval`)

Вызывается автоматически при регистрации акта (если `APPROVAL_MODE` не `false`). Порядок:

1. `gu23_approval_init` — создаёт `pending`-записи в `xx_disl_gu23_approval`
2. `gu23_approval_get_signers` — получает подписантов предприятия с учётными записями
3. Для каждого генерирует HMAC-ссылку, сохраняет `token_sig` в БД
4. Отправляет письмо (режим определяется переменной `APPROVAL_MODE`)

### Режим рассылки (`APPROVAL_MODE` в `form.js`)

```js
// 'send_mail' — реальная отправка через PHP mail()
// 'send_file' — сохранить HTML-письмо в папку mail/ (для тестирования)
// false        — не отправлять ничего
const APPROVAL_MODE = 'send_file'
```

Значение передаётся в POST-параметре `mode` при вызове `gu23_send_approval`. PHP использует его для выбора между `mail()` и `saveMailToFile()`.

### Подписание через приложение (`gu23_approve_in_app`)

Подписант с ролью `isUserSigner` может подписать/отклонить акт прямо в интерфейсе (без перехода по HMAC-ссылке). При подписании пишется запись в `xx_disl_gu23_hist`.

### Подписание по HMAC-ссылке

Ссылки вида `/gu23/approve.php?act=…&uid=…&action=approve&sig=…` обрабатываются `ApprovalRepository.php`. Срок действия — 7 дней.

## Установка и миграции

### Первичная установка

```sql
-- 1. DDL основных таблиц и представления
@sql-DB/install_gu23_all.sql

-- 2. DDL таблицы согласований
@sql-DB/install_approval.sql

-- 3. Компиляция пакета
@sql-DB/XX_DISL_GU23_PKG.pks
@sql-DB/XX_DISL_GU23_PKG.pkb
```

### Миграции (обновление существующей базы)

```sql
-- Добавить поле user_id в справочник подписантов РЖД
@sql-DB/alter_ref_signer_user_id.sql

-- Добавить поле stype в таблицу подписантов акта (различение own/rzd)
@sql-DB/migrate_signer_stype.sql

-- Перекомпилировать пакет после изменений
@sql-DB/XX_DISL_GU23_PKG.pks
@sql-DB/XX_DISL_GU23_PKG.pkb
```

> **Важно:** backfill в `migrate_signer_stype.sql` определяет `stype` по наличию записи в `xx_disl_users`. Для старых актов, где ID в `xx_disl_users` и `xx_disl_gu23_ref_signer` совпадают, результат может быть неточным. Для новых актов `stype` передаётся фронтендом и сохраняется точно.

## Локальная разработка

Папки `test/` и `mail/` добавлены в `.gitignore`.

- `mail/` — HTML-файлы писем при `APPROVAL_MODE = 'send_file'`
- `test/` — вспомогательные тестовые скрипты

Для отключения рассылки при разработке установить `APPROVAL_MODE = false` в `form.js`.
