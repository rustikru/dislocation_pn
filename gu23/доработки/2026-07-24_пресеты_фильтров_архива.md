# Доработка: пресеты фильтров на странице «Архив актов»

**Дата:** 2026-07-24
**Автор задачи:** BekmansurovRR
**Область:** только папка `gu23`
**СУБД:** Oracle 11 (нет нативного JSON)

## Проблема / цель

На странице «Архив актов» много фильтров (тип, статус, цех, категория причины,
период, доп. фильтры). Пользователь вынужден каждый раз выставлять их заново.
Хотим, чтобы каждый цех/руководитель мог настроить «свой контроль»:

- пользователь **сохраняет именованные пресеты** отбора;
- может назначить **пресет по умолчанию** (применяется автоматически при
  открытии страницы);
- (как частный случай «последней комбинации» — отдельный служебный пресет,
  см. «Расширения»).

## Подход

Хранение — **в таблице БД** (переносится между устройствами, в отличие от
localStorage). Параметры отбора храним **одним полем в формате JSON (CLOB)** —
БД его не разбирает, парсят PHP и фронт. Это гибко: при добавлении новых
фильтров структура таблицы не меняется.

## Модель данных (Oracle 11)

```sql
create table xx_disl_gu23_filter_preset (
   id         number not null,
   user_id    number not null,                 -- владелец пресета
   name       varchar2(128) not null,          -- наименование пресета
   params     clob not null,                   -- параметры отбора (JSON-текст)
   is_default varchar2(1) default 'N' not null, -- применять по умолчанию
   created_at date default sysdate not null,
   constraint xx_disl_gu23_filter_preset_pk primary key ( id ),
   constraint xx_disl_gu23_filter_preset_uk unique ( user_id, name ),
   constraint xx_disl_gu23_filter_preset_ck check ( is_default in ( 'Y', 'N' ) )
);

create sequence xx_disl_gu23_filter_preset_seq;
```

Особенности Oracle 11:
- НЕ используем `check (params is json)` — валидация JSON на фронте/в PHP.
- PK через sequence (не identity): `id := xx_disl_gu23_filter_preset_seq.nextval`
  при insert (как в остальных таблицах проекта), триггер не нужен.
- Уникальность `(user_id, name)` — имена пресетов уникальны в рамках юзера,
  сохранение = upsert (`MERGE`).
- Уникальность на `params` (CLOB) не вешаем (и не нужна).
- «Пресет по умолчанию» — на пользователя один: при установке нового сбрасывать
  прежний (`update ... set is_default='N' where user_id=? and is_default='Y'`).

## Что кладём в params (JSON)

Ровно нужные поля объекта `archiveFilter` из `gu23/js/registry.js`, БЕЗ `page`.
Пример:
```json
{"type":"start,end","status":"active","dept":"12","reason_categ":"","has_signed":"","period":"month"}
```
- `type/status/dept/reason_categ` — строки значений «через запятую» (как сейчас
  в `archiveFilter[key]`).
- `has_signed` — '' или 'Y'.
- `period` — ОТНОСИТЕЛЬНЫЙ период (`month` / `last30` / `all` / …), а не
  абсолютные даты. Абсолютные `date_from/date_to` в пресет НЕ пишем, иначе
  пресет «застынет» на конкретном месяце; конкретные даты вычисляем при
  применении. (Как вариант — период в пресет не включать вовсе, тогда даты
  всегда = текущий месяц.)

## Пакет `XX_DISL_GU23_PKG` (сигнатуры)

- `gu23_filter_preset_save(p_user_id in number, p_name in varchar2, p_params in clob, p_is_default in varchar2 default 'N') return varchar2`
  — `MERGE` по `(user_id, name)`; при `p_is_default='Y'` сбросить прежний
  default этого пользователя. Возвращает 'OK' / 'ERR<us>...'.
- `gu23_filter_presets_get(p_user_id in number) return <tab> pipelined`
  — список пресетов пользователя. Поле `params` отдавать как ТЕКСТ через
  `dbms_lob.substr(params, 4000, 1)` (набор фильтров маленький, влезает в 4000),
  чтобы PHP получил обычный varchar2 без возни с LOB-дескриптором.
- `gu23_filter_preset_delete(p_user_id in number, p_id in number) return varchar2`
  — удаление своего пресета (проверять `user_id`).
- (опц.) `gu23_filter_preset_set_default(p_user_id, p_id) return varchar2`
  — переключение default без пересохранения params.

Новый row-тип, напр. `t_gu23_filter_preset_row (id number, name varchar2(128),
params varchar2(4000), is_default varchar2(1))` + tab-тип, в спецификации `.pks`.

## Чтение/запись CLOB (паттерн уже есть в проекте)

- **Запись:** как `p_signers` в `gu23_save_act` — PHP `putClobParam($st, ':params', $json)`.
- **Чтение:** отдавать `dbms_lob.substr(params,4000,1)` (см. выше) — тогда
  `selectRows()` вернёт строку, и `json_decode`/`JSON.parse` работают без
  обработки LOB.

## PHP (`gu23/classes/GuActRepository.php`)

Новые `ajax_action` + приватные методы:
- `gu23_filter_preset_save` — принять `name`, `params` (JSON-строка), `is_default`;
  `params` биндить через `putClobParam`.
- `gu23_filter_presets_get` — вернуть список пресетов текущего пользователя.
- `gu23_filter_preset_delete` — по `id`.
`user_id` всегда брать из `$this->auth->getUserId()` (не из запроса).

## Фронт (`gu23/js/registry.js`)

Опорные места: объект `archiveFilter` (`:22`), построение фильтров
`addMultiChoiceFilter` (`:53`), даты/доп.фильтры, `loadArchiveData()`,
кнопка «Сбросить» (`:230`).

Нужно:
1. **Селектор пресетов** рядом со «Сбросить» (компактно — выпадающий список +
   «kebab» ⋮ с действиями «Сохранить как пресет…», «Сделать по умолчанию»,
   «Удалить»). Не добавлять много кнопок в ряд.
2. **Применение пресета:** `JSON.parse(params)` → `Object.assign(archiveFilter, parsed)`
   → вычислить даты из `period` → **преднастроить виджеты** (сейчас они всегда
   строятся пустыми, `:80–98`): проставить `checked` чекбоксам из
   `archiveFilter[key]`, вызвать `refreshFilterButton()`; выставить даты и
   `has_signed`; → `loadArchiveData()`.
3. **Сохранение:** собрать `archiveFilter` (нужные поля) → `JSON.stringify` →
   `gu23_filter_preset_save`.
4. **По умолчанию при загрузке:** в `showArchive` после получения списка
   пресетов — если есть `is_default='Y'`, применить его; иначе системные
   дефолты (текущий месяц).
5. **«Сбросить»** — вернуть системные дефолты (пресет при этом не удалять).

## Round-trip (кратко)
Сохранение: `archiveFilter` → `JSON.stringify` → PHP(`putClobParam`) → пакет(`MERGE`) → CLOB.
Применение: пакет(`dbms_lob.substr` → текст) → PHP → JS `JSON.parse` → `archiveFilter` → виджеты → перезагрузка.
Вся «магия параметров» — это `stringify`/`parse`; таблица хранит непрозрачный текст.

## Объём (итого)
- 1 таблица + 1 sequence.
- 3–4 функции в пакете (+ 1 row/tab тип в `.pks`).
- 2–3 `ajax_action` в PHP.
- Селектор + сохранение/применение в `registry.js` (+ преднастройка виджетов).

## Расширения (на подумать — обсуждаемо)
- **«Последняя комбинация»** — как служебный пресет с зарезервированным именем
  (напр. `__last__`), обновляемый автоматически в `loadArchiveData()`; при
  загрузке применять его, если нет явного default.
- **Общие пресеты на цех/роль** — добавить в таблицу `scope` (`user`/`dept`/`role`)
  и `scope_id`; тогда руководитель может раздать «контроль» цеху. Потребует
  правил видимости в `gu23_filter_presets_get`.
- **Порядок/избранные пресеты** — поле `ord_no`/`is_pinned`.
- **Экспорт/импорт пресета** — раз params это JSON, можно копировать между
  пользователями.
- **Переиспользование под другие страницы** — обобщить до
  `xx_disl_gu23_user_pref(user_id, pref_key, pref_val clob)` и хранить пресеты
  под ключом `archive_filters`. Плюс — единое хранилище настроек; минус — чуть
  абстрактнее.
