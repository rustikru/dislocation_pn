# ГУ-23. Бизнес-логика оформления актов

Документ описывает, как заводятся, согласуются, закрываются и аннулируются
акты общей формы (ГУ-23) в модуле. Технические ограничения по данным вынесены
в конец как справочник.

---

## 1. Типы актов

| Тип (`act_type`) | Назначение | Особенности оформления |
|---|---|---|
| `start` (Начало простоя) | Фиксирует **начало** простоя вагонов | Дата начала, вагоны/груз, причина |
| `end` (Окончание простоя) | Закрывает простой по акту начала | Привязан к акту начала, дата окончания, расчёт длительности |
| `other` (Прочий акт) | Разовый акт общей формы | Без привязки к началу/окончанию; подписантов достаточно 2 |

Ключевая связка — **`start` → `end`**: акт окончания всегда строится на основе
конкретного акта начала (`linked_start_id`) и закрывает именно его вагоны.

---

## 2. Жизненный цикл акта (статусы)

```
        создание
           │
        ┌──▼──┐   отправка на        все подписали
        │draft│──подписание────►active──────────────►signed
        └──┬──┘                   │  │                  │
           │ удаление             │  │ один отклонил    │ закрытие
           ▼                      │  ▼                  │ (только end)
        (удалён)                  │ rejected            ▼
                                  │                   closed
                                  └──────► annulled ◄───┘
                                          (аннулирование)
```

- **draft (Проект)** — черновик. Только в этом статусе акт можно **редактировать** и **удалять**.
- **active (Открыт)** — отправлен на подписание; идёт сбор подписей.
- **signed (Подписан)** — все подписанты предприятия одобрили.
- **rejected (Отклонён)** — хотя бы один подписант отклонил (маршрут останавливается).
- **closed (Закрыт)** — для акта окончания: завершён; для акта начала: все вагоны закрыты актами окончания.
- **annulled (Аннулирован)** — отменён с указанием причины.

> Действующий или закрытый акт **не редактируется**. Если нужно изменить — акт
> аннулируют и заводят новый.

---

## 3. Оформление акта начала простоя (`start`)

1. Указывается **цех** (по нему формируется номер акта), станция составления,
   станция отправления и назначения, причина и обстоятельства.
2. Вагоны подгружаются **из дислокации** (по номерам вагонов и/или номеру
   накладной). Для каждого вагона сохраняется его собственная **накладная**,
   собственник, род, маршрут, груз, вес.
3. Можно завести акт «по грузу/накладной» без перечня вагонов — тогда хотя бы
   одно из полей (вагоны / груз / накладная) должно быть заполнено.

### Запрет повторного простоя по одним и тем же данным

Чтобы один и тот же вагон/накладная не попали в два простоя одновременно:

- **По вагону** — нельзя завести новый акт начала, если в **том же
  календарном месяце** уже есть акт начала с этим вагоном.
- **По накладной** — нельзя, если в окне **±3 месяца** уже есть акт начала с
  этой накладной.

«Занятым» считается акт начала в статусе `active` / `signed` / `closed`.
Статусы `annulled` / `rejected` / `draft` **освобождают** вагон/накладную.

Проверка работает на двух уровнях:
- **рано** — при подгрузке из дислокации занятые вагоны/накладные не
  добавляются в таблицу, а показываются как пропущенные с номером
  существующего акта;
- **жёстко** — повторно при сохранении на подписание (финальный барьер,
  считается от фактической даты начала акта). Обходится только техническим
  флагом `force`.

---

## 4. Оформление акта окончания простоя (`end`)

1. Акт окончания строится **на основе открытого акта начала**. В список выбора
   попадают только акты начала, у которых **остались незакрытые вагоны**
   (полностью закрытые акты начала из списка исчезают).
2. Вагоны и их данные берутся из акта начала (не из дислокации).
3. Указывается **дата окончания**; она должна быть **≥ даты начала** и **не в
   будущем**. По разнице дат рассчитываются длительности (часы, сутки,
   календарные сутки).

### Частичное и полное закрытие (по вагонам)

- Один акт начала можно закрывать **по частям** — несколькими актами
  окончания на разные вагоны.
- Каждый вагон в акте окончания должен **относиться** к выбранному акту начала.
- Один вагон **нельзя закрыть дважды**: если он уже покрыт другим актом
  окончания (`active`/`signed`/`closed`), система не даст добавить его повторно.
- Когда закрыты **все** вагоны акта начала, акт начала автоматически переходит
  в статус **`closed`**.

> Именно эта логика «покрытия» не даёт ситуации «64 вагона в начале — 65 в
> окончаниях»: подписанные и закрытые акты окончания тоже занимают вагон.

---

## 5. Согласование (подписание)

- Маршрут согласования сейчас **параллельный**: всем подписантам предприятия
  (`stype='own'`) одновременно отправляются запросы.
- Подписать акт предприятия должны **все** такие подписанты — тогда акт
  становится **`signed`**.
- **Любое отклонение** переводит акт в **`rejected`** и останавливает маршрут;
  причина отклонения сохраняется и дублируется в карточку акта.
- Один подписант = одно решение на акт; повторное решение перезаписывает
  предыдущее.
- Требуемое число подписантов в форме: **3** для `start`/`end`, **2** для
  `other`. Подписант, введённый вручную, считается заполненным только с
  должностью и организацией.
- Ссылки на подписание подписываются HMAC (без хранения токенов в БД);
  повторная отправка не инвалидирует ранее выданную ссылку.

---

## 6. Закрытие и аннулирование

### Закрытие (`closed`)
- Закрывать вручную можно **только акт окончания** и только из статуса
  `active` или `signed`.

### Аннулирование (`annulled`) — с обязательной причиной
- Аннулирование **акта окончания** → связанный акт начала, если был `closed`,
  **снова открывается** (`active`): его вагоны опять свободны для закрытия.
- Аннулирование **акта начала** → **каскадно** аннулируются все связанные акты
  окончания (кроме уже `annulled`/`draft`).

## 7. Доступ (полномочия)

Действия ограничены правами роли:

| Действие | Право |
|---|---|
| Создание/правка акта | `CREATE_ACT`, `EDIT_OWN_ACT` |
| Отправка на подписание / переотправка ссылки | `SEND_APPROVAL` |
| Подписание из интерфейса | `SIGN_ACT` |
| Аннулирование | `ANNUL_ACT` |
| Закрытие акта | `CLOSE_ACT` |
| Просмотр всех актов | `VIEW_ALL_ACTS` |
| Справочники | `MANAGE_REFS` |
| Роли и полномочия | `MANAGE_ROLES` |

Вход в модуль — наличие роли (`gu23_can_access`). В DEV-режиме (есть
`db_config.local.php`) проверки прав обходятся.

## Приложение. Ограничения уровня данных (справочник)

**Домены значений:**
- `act_type` ∈ `start` / `end` / `other`
- `status` ∈ `draft` / `active` / `signed` / `closed` / `annulled` / `rejected`
- `signer.stype` ∈ `own` / `rzd` / null (вручную)
- `ref_reason.act_kind` ∈ `start` / `end` / `other` / `any`
- `active` ∈ `'Y'` / `'N'`

**Обязательные поля (NOT NULL):** `act.act_type`, `act.status`;
`act_row.act_id`, `act_row.wagon_no`; `act_id` у файлов/подписантов/истории;
`ref_reason.name`, `ref_signer.fio`.

**Длины полей:** `wagon_no` 16; `waybill_no` 64; `owner/kind/st_from/st_to` 128;
`cargo` 256; `reason` 512; `circumstances` 4000; `annul_reason` 1000;
`act_number` 64; коды станций 50.

**Связи/целостность:**
- Вагоны, файлы, подписанты, история ссылаются на акт через FK
  `ON DELETE CASCADE`.
- `counter` UNIQUE `(dept_id, yr)` — один счётчик номеров на цех/год.
- Номер акта уникален в пределах цех + год.
- Роли/права: `role_code`, `perm_code` уникальны; назначения через составные PK
  (повторно не назначить).
