# Модуль ГУ-23

Этот файл описывает, как устроен модуль ГУ-23, где лежат основные части, как создается акт, как он редактируется, подписывается, печатается и какие ошибки чаще всего встречаются.

Документ написан для сопровождения. Если нужно быстро понять, где искать нужную логику, начинай с разделов `Точки входа`, `Package` и `Типовые ошибки`.

## Коротко о модуле

Модуль ГУ-23 ведет акты общей формы:

- акт на начало простоя;
- акт на окончание простоя;
- прочий акт.

Пользователь работает через страницу `gu23/index.php`. Карточку конкретного акта можно открыть напрямую через `gu23/card.php?id=...`.

Основная бизнес-логика находится в Oracle package `XX_ETW.XX_DISL_GU23_PKG`. PHP не должен напрямую менять таблицы модуля. PHP вызывает package, принимает результат и показывает данные на странице.

## Как строится страница

1. Пользователь открывает `gu23/index.php`.
2. PHP проверяет авторизацию и доступ к модулю через package.
3. Загружается основной HTML-каркас: левое меню и область `#view`.
4. `js/app.js` вызывает `gu23_get_refs`, получает справочники и права.
5. По текущему URL выбирается страница:
   - `index.php?page=archive` - архив;
   - `index.php?page=new` - создание;
   - `card.php?id=...` - карточка акта;
   - `index.php?page=refs` - справочники;
   - `index.php?page=roles` - роли.
6. JS загружает PHP-шаблон из `gu23/pages` и заполняет его данными.
7. Все действия идут через `gu23/data.php` с параметром `action`.

## Разделы модуля

### Создать акт

Форма создания находится в `pages/form.php`, поведение формы - в `js/form.js`.

В форме выбирается тип акта:

- `start` - начало простоя;
- `end` - окончание простоя;
- `other` - прочий акт.

Основные особенности:

- для начала простоя обязательно указывается дата начала;
- для окончания простоя обязательно выбирается связанный акт начала простоя;
- для окончания простоя дата окончания не может быть раньше даты начала;
- для начала и прочего акта обязателен груз;
- обстоятельства обязательны;
- должен быть хотя бы один вагон;
- подписанты проверяются по настройке типа акта.

Вагоны можно ввести вручную и подтянуть данные из дислокации кнопкой `Заполнить данные из Дислокации`.

### Архив актов

Раздел показывает список актов. Логика находится в:

- `pages/archive.php`;
- `js/registry.js`;
- package-функции `gu23_get_acts` и `gu23_count_acts`.

Сейчас на страницу выводится 20 записей.

Фильтры передаются в package:

- строка поиска;
- тип акта;
- статус;
- цех;
- дата с;
- дата по;
- признак наличия подписанных файлов;
- номер страницы.

Сортировка актов задается в package. Если нужно менять порядок вывода, сначала смотреть `gu23_get_acts`.

### Карточка акта

Карточка открывается по `gu23/card.php?id=...`.

В карточке показываются:

- реквизиты акта;
- вагоны;
- подписанты;
- приложения;
- история.

Карточка получает данные через `gu23_get_act`, а PHP собирает полный ответ в `GuActRepository::getActCard`.

### Справочники

Раздел находится в:

- `pages/refs.php`;
- `js/refs.js`;
- package-функции `gu23_ref_signers_all`, `gu23_ref_reasons_all`, `gu23_ref_signer_save`, `gu23_ref_reason_save` и похожие.

Справочники доступны всем пользователям для просмотра.

Добавление, редактирование и включение/отключение записей доступны только пользователям с правом `MANAGE_REFS`.

Справочник `Причины` поддерживает:

- поиск по названию;
- фильтр по типу акта;
- фильтр по категории;
- фильтр по статусу;
- выгрузку в Excel с учетом активных фильтров.

Категории причин берутся из общего справочника `XX_DISL_GENERAL_REF` по `REF_CODE = 'CATEG_CAUSE'`.

Типы актов для справочника причин берутся из `XX_DISL_GENERAL_REF` по `REF_CODE = 'GU23_ACT_KIND'`. Если справочник пустой, PHP использует запасные значения:

- `start` - Начало;
- `end` - Окончание;
- `other` - Прочий;
- `any` - Любой.

### Роли

Раздел находится в:

- `pages/roles.php`;
- `js/roles.js`;
- package-функциях `gu23_roles_get_all`, `gu23_users_roles_get`, `gu23_role_assign`, `gu23_role_revoke`, `gu23_role_perms_get`.

Редактирование ролей доступно только пользователям с правом `MANAGE_ROLES`.

### Подписание по ссылке

Страница `approve.php` открывается по ссылке из письма. Она работает по токену.

Пользователь может:

- подписать;
- отклонить;
- отправить в корректировку.

Если акт уже закрыт или аннулирован, действие по ссылке запрещается.

### Отчеты

Отчет открывается через:

```text
gu23/report/report.php?id=123&format=docx
gu23/report/report.php?id=123&format=pdf
```

DOCX формируется по шаблону Word.

PDF формируется из DOCX через LibreOffice (`soffice`). Для этого на сервере должен быть доступен `soffice` или должен быть указан путь в `gu23/config.php`.

PDF открывается в браузере через `inline`, а не принудительно скачивается.

## Типы актов

| Код | Название | Назначение |
| --- | --- | --- |
| `start` | Начало простоя | Фиксирует начало простоя вагона. |
| `end` | Окончание простоя | Закрывает простой по ранее созданному акту начала. |
| `other` | Прочий акт | Используется для прочих случаев по ГУ-23. |
| `any` | Любой | Используется в справочнике причин, если причина подходит для любого типа акта. |

## Статусы акта

| Статус | Что означает |
| --- | --- |
| `draft` | Проект. Акт еще не отправлен на подписание. |
| `active` | Открыт / находится на подписании. |
| `on_correction` | Отправлен на корректировку создателю. |
| `signed` | Подписан. Статус встречается в старой логике и учитывается для некоторых отчетов. |
| `closed` | Закрыт. Сейчас после подписания всеми обязательными подписантами акт должен стать закрытым. |
| `annulled` | Аннулирован. |

## Создание акта

### Общий путь создания

1. Пользователь открывает форму создания.
2. JS создает черновик на стороне браузера.
3. Пользователь заполняет поля.
4. JS проверяет обязательные поля.
5. При сохранении вызывается `gu23_save_act`.
6. PHP передает данные в package.
7. Package создает заголовок акта, строки вагонов, подписантов и запись истории.
8. Если пользователь выбрал сохранение с отправкой на подписание:
   - акт получает статус `active`;
   - создается маршрут согласования;
   - письмо отправляется первому подписанту.

### Особенности акта начала простоя

Для начала простоя обязательны:

- цех;
- станция составления;
- станция отправления;
- станция назначения;
- груз;
- причина;
- начало простоя;
- обстоятельства;
- вагоны;
- обязательные подписанты.

После создания такой акт считается открытым началом простоя для вагонов.

### Особенности акта окончания простоя

Для окончания простоя обязательно выбирается связанный акт начала простоя.

Обычно строки вагонов подтягиваются из открытого начала простоя.

При сохранении окончания package связывает строки окончания с началом. Если все вагоны по акту начала закрыты окончаниями, связанный акт начала тоже может закрываться.

### Особенности прочего акта

Прочий акт не закрывает простой. Он живет как самостоятельный акт.

Для прочего акта используется отдельная настройка подписантов:

- несколько подписантов от предприятия;
- подписант РЖД может быть необязательным, если так настроен шаблон и форма.

## Редактирование акта

Редактировать акт можно не всегда.

Основное правило:

- проект может редактировать создатель акта или администратор;
- акт на корректировке может редактировать создатель акта или администратор;
- закрытые и аннулированные акты редактировать нельзя.

Проверка находится в package-функции `gu23_can_edit_draft`.

При повторной отправке акта из статуса `on_correction`:

- статус снова становится `active`;
- версия документа увеличивается;
- маршрут согласования сбрасывается;
- согласование начинается заново с первого подписанта.

## Подписание и маршрут согласования

Согласование идет по очереди.

Порядок такой:

1. При отправке акта вызывается `gu23_approval_init`.
2. Package создает записи подписантов в `XX_DISL_GU23_APPROVAL`.
3. Затем вызывается `gu23_send_next_approval_mail`.
4. Письмо получает только первый ожидающий подписант.
5. После подписи первого подписанта отправляется письмо следующему.
6. Если кто-то отклонил акт, дальше письма не отправляются.
7. Если кто-то отправил акт в корректировку, письмо уходит создателю акта.
8. Когда все нужные подписанты подписали, акт становится `closed`.

### Подписание из интерфейса

Подписание из карточки идет через `gu23_direct_decision`.

Такой вариант используется, когда подписант уже находится в модуле и нажимает кнопку прямо в карточке.

### Подписание по ссылке из письма

Подписание по ссылке идет через `approve.php`.

В этом случае используется токен, который хранится в `XX_DISL_GU23_APPROVAL.TOKEN_SIG`.

Решение сохраняется через `gu23_approval_save_decision`.

### Отправка в корректировку

Статус корректировки: `on_correction`.

При отправке в корректировку:

- акт получает статус `on_correction`;
- создателю акта отправляется письмо;
- причина корректировки сохраняется в истории и в записи согласования;
- после исправления акт отправляется на согласование заново.

### Ручные подписанты

Если подписант введен вручную и не связан с пользователем системы, у него нет электронной подписи.

Такой подписант должен выводиться в обычном блоке подписей в отчете.

В ПЭП-блок попадают только подписанты, у которых есть пользователь системы и решение `approved`.

## Права

Основные права задаются в БД и проверяются через package.

| Право | Для чего нужно |
| --- | --- |
| `CREATE_ACT` | Создание акта. |
| `EDIT_OWN_ACT` | Редактирование своего проекта или акта на корректировке. |
| `DELETE_ACT` | Удаление проекта. |
| `ANNUL_ACT` | Аннулирование акта. |
| `CLOSE_ACT` | Ручное закрытие акта. |
| `SEND_APPROVAL` | Отправка акта на согласование. |
| `SIGN_ACT` | Подписание акта из интерфейса. |
| `MANAGE_REFS` | Редактирование справочников. |
| `MANAGE_ROLES` | Управление ролями и правами. |

Администратор определяется через `gu23_is_admin`. Обычно это роль администратора модуля.

Кнопка повторной рассылки ссылки на подписание дополнительно ограничена: она доступна только администратору или пользователю `USER1`.

## Приложения к акту

Файлы хранятся на диске в папке:

```text
gu23/act_data/{тип_акта}/{id_акта}/
```

В БД хранится путь к файлу относительно модуля, а не абсолютный путь сервера. Это нужно, чтобы при смене корня сайта или имени сервера файлы не ломались.

Категории файлов:

- `general` - общие приложения;
- `signed` - подписанные файлы.

Правила:

- возможность добавления и удаления проверяется через package;
- подписанные файлы удаляет только администратор;
- для закрытых актов удаление ограничено строже;
- аннулированные акты лучше считать недоступными для изменения файлов.

## Отчеты DOCX и PDF

### Как формируется отчет

1. `report.php` получает `id` и `format`.
2. Через package загружаются акт, вагоны, подписанты и согласования.
3. `GuActPhpWordReport.php` выбирает шаблон:
   - `act23_start.docx`;
   - `act23_end.docx`;
   - `act23_general.docx`.
4. В шаблоне заменяются плейсхолдеры.
5. Для DOCX файл отдается сразу.
6. Для PDF DOCX сначала конвертируется через LibreOffice.

### Основные плейсхолдеры

| Плейсхолдер | Что выводит |
| --- | --- |
| `{ACT_NUMBER}` | Номер акта. |
| `{DEPT}` | Цех. |
| `{STATION}` | Станция составления. |
| `{ST_FROM}` | Станция отправления. |
| `{ST_TO}` | Станция назначения. |
| `{REASON_NAME}` | Причина. |
| `{CARGO_REF}` | Груз. |
| `{START_AT}` | Начало простоя или дата акта. |
| `{END_AT}` | Окончание простоя. |
| `{CREATED_AT}` | Дата создания. |
| `{CIRCUMSTANCES}` | Обстоятельства. |
| `{SIGNER_LINE}` | Верхний список лиц, в присутствии которых составлен акт. |
| `{SIGNER_POST}` | Должность подписанта в блоке подписи. |
| `{SIGNER_FIO}` | ФИО подписанта в блоке подписи. |
| `{SIGNER_RZD_POST}` | Должность участника РЖД. |
| `{SIGNER_RZD_FIO}` | ФИО участника РЖД. |
| `{PEP_FULL_NAME}` | ФИО подписанта в ПЭП-блоке. |
| `{PEP_APPROVED_ID}` | Идентификатор пользователя в ПЭП-блоке. |
| `{PEP_DECIDED_AT}` | Дата подписания в ПЭП-блоке. |

Для таблицы вагонов используются плейсхолдеры:

### Когда выводится ПЭП-блок

ПЭП-блок выводится, когда документ считается подписанным всеми сторонами.

Условия:

- по акту есть согласования;
- все нужные согласования имеют статус `approved`;
- у подписантов есть `USER_ID`;
- для начала простоя допускается статус `signed` или `closed`;
- для остальных типов обычно нужен статус `closed`.

Если подписант ручной, он не попадает в ПЭП-блок и остается в обычной подписи.

### Как PHP получает результат из package

Для многих процедур используется строковый ответ в формате:

Для передачи списков из PHP в package используются CLOB-строки:

## Характерные ошибки и что делать

### `ORA-04063: package body "XX_ETW.XX_DISL_GU23_PKG" has errors`

Package скомпилировался с ошибками.

Проверить ошибки:

```sql
select line, position, text
from user_errors
where name = 'XX_DISL_GU23_PKG'
order by sequence;
```

Исправить строки из `XX_DISL_GU23_PKG.pkb` и перекомпилировать package body.

### `PLS-00231: функция 'G_USER_NAME' не может быть использована в SQL`

Внутренняя PL/SQL-функция вызвана прямо внутри SQL-запроса.

Как исправлять:

1. Вызвать функцию заранее в PL/SQL-переменную.
2. Эту переменную использовать в `insert` или `update`.

Пример:

```sql
v_user_name := g_user_name(p_user_id);
```

И дальше:

```sql
insert into ... values (..., v_user_name, ...);
```

### `Ajax ошибка ... JSON Parse error: Unrecognized token '<'`

Браузер ждал JSON, а получил HTML.

Обычно причина:

- PHP warning;
- fatal error;
- Oracle exception;
- лишний вывод до JSON;
- ошибка авторизации, которая вернула HTML-страницу.

Что смотреть:

1. `gu23/log/gu23-YYYY-MM-DD.log`;
2. вкладку Network в браузере;
3. ответ `gu23/data.php`;
4. ошибки PHP на сервере.

### `LibreOffice не найден для формирования PDF`

PDF создается через `soffice`. Нужно установить LibreOffice или указать путь.

Проверка на сервере:

```bash
command -v soffice
soffice --version
```

Настройка в `gu23/config.php`:

```php
'soffice_path' => '/usr/bin/soffice',
```

### `Composer dependencies require a PHP version >= 8.3.0`

На сервере PHP 8.1, а папка `vendor` собрана под PHP 8.3.

Что делать:

1. На машине с PHP 8.1 или через composer platform собрать зависимости под PHP 8.1.
2. Обновить `composer.lock`.
3. Залить подходящий `vendor` на сервер, если на сервере нет интернета.

### `DOCX сформирован с ошибкой XML: StartTag: invalid element name`

DOCX внутри является ZIP-архивом с XML. Ошибка значит, что в XML попал неправильный символ или сломан тег.

Частые причины:

- в данные попали символы `<`, `>`, `&` без экранирования;
- плейсхолдер в шаблоне сломан;
- таблица или колонтитул в шаблоне повреждены Word-ом;
- ручная замена XML нарушила структуру документа.

Что делать:

1. Проверить данные акта: обстоятельства, груз, собственник, причина, должности подписантов.
2. Проверить, что значения перед вставкой проходят экранирование.
3. Открыть шаблон и пересохранить его в Word.
4. Проверить проблемный `document.xml` по строке и колонке из ошибки.

### `Подписант не найден`

Обычно значит, что package не нашел текущего подписанта для отправки письма или решения.

Проверить:

- есть ли записи в `XX_DISL_GU23_APPROVAL`;
- есть ли у подписанта `USER_ID`;
- есть ли email;
- совпадает ли подписант с текущим ожидающим подписантом;
- не подписан ли акт уже другим человеком;
- не сбит ли порядок `ORD_NO`.

### `Ожидается другой подписант`

Маршрут согласования последовательный.

Подписывать может только тот пользователь, который сейчас первый в очереди со статусом `pending`.

### `В акте нет вагонов`

Акт нельзя отправить на согласование без строк вагонов.

Проверить:

- заполнено ли поле вагонов в форме;
- нажималась ли кнопка подтягивания данных из дислокации;
- сохранились ли строки в `XX_DISL_GU23_ACT_ROW`.

### Не видны кнопки

Кнопки зависят от прав, статуса акта и текущего пользователя.

Проверить:

```sql
select *
from table(xx_disl_gu23_pkg.gu23_user_perms_get(:user_id));
```

Также проверить:

- `$_SESSION` через `gu23/session_debug.php`;
- является ли пользователь администратором;
- является ли пользователь создателем акта;
- является ли пользователь текущим подписантом;
- не закэшировался ли старый JS.

### IP подписанта пустой

Перед действиями PHP должен вызвать:

```sql
xx_disl_gu23_pkg.gu23_set_client_ip(:ip)
```

Также решение должно передавать IP в package. Смотреть `approve.php`, `GuActRepository::approveInApp` и функции `gu23_approval_save_decision`, `gu23_direct_decision`.

### Неверная дата или ошибка "дата в будущем"

Проверить дату и часовой пояс Oracle:

```sql
select
  to_char(sysdate, 'YYYY-MM-DD HH24:MI:SS') sysdate_val,
  to_char(systimestamp, 'YYYY-MM-DD HH24:MI:SS TZH:TZM') systs,
  sessiontimezone,
  dbtimezone
from dual;
```

Если Oracle в Docker, отдельно проверить время контейнера:

```bash
docker exec disl_oracle date
docker exec disl_oracle sh -lc 'date +%Z%z'
```

### Кириллица в имени файла отображается неправильно

Проверить заголовок `Content-Disposition` в `get_file.php`.

Для кириллицы нужно отдавать имя через `filename*` в UTF-8.

### Письмо не отправилось

Проверить:

- email подписанта;
- `gu23_send_mail`;
- настройки отправки почты;
- таблицу отладки писем, если она используется;
- результат `gu23_send_next_approval_mail`;
- лог `gu23/log/gu23-YYYY-MM-DD.log`.

## Проверки после изменений

### Проверить PHP-синтаксис

```bash
php -l gu23/data.php
php -l gu23/classes/GuActRepository.php
php -l gu23/report/GuActPhpWordReport.php
```

### Проверить package после компиляции

```sql
select name, type, line, position, text
from user_errors
where name = 'XX_DISL_GU23_PKG'
order by type, sequence;
```

### Проверить основной сценарий

1. Создать проект.
2. Отредактировать проект.
3. Отправить на согласование.
4. Проверить, что письмо ушло первому подписанту.
5. Подписать первым.
6. Проверить, что письмо ушло следующему.
7. Подписать всеми.
8. Проверить, что акт стал `closed`.
9. Сформировать DOCX.
10. Сформировать PDF.
11. Открыть отчет и проверить подписи, ПЭП-блок и таблицу вагонов.

### Проверить корректировку

1. Отправить акт на согласование.
2. Текущим подписантом нажать `В корректировку`.
3. Проверить, что акт стал `on_correction`.
4. Проверить письмо создателю.
5. Создателем изменить акт.
6. Повторно отправить на согласование.
7. Проверить, что версия документа увеличилась и маршрут начался сначала.

## Правила сопровождения

1. Бизнес-логику менять в package, а не прямыми SQL-запросами в PHP.
2. PHP должен быть тонкой прослойкой: принять запрос, вызвать package, вернуть JSON или файл.
3. Не добавлять прямые обращения к таблицам из PHP, если это не простая служебная проверка и ее нельзя иначе сделать.
4. При изменении package обновлять и `.pks`, и `.pkb`, если меняется публичная функция.
5. После изменения шаблона Word обязательно проверять и DOCX, и PDF.
6. Не использовать сложные названия функций без необходимости.
7. Не коммитить `gu23/act_data`, `gu23/log`, `vendor`.
8. Если меняется структура таблицы, сразу обновить этот README.
